## Anti-Patterns Verdict

**Does this look AI-generated?** Partial.

**LLM assessment:** The tonal near-black stack, hairline borders, flat session feed, earned status colors, and permission strip are disciplined product UI — not cream SaaS or metric-hero cliché. A fluent product designer would trust the approval row and quiet idle bar. They would **pause hard** at the multicolor lasers (`.notch-laser*` + `.session-laser*`) on the bar and every working card: rainbow gradient, glow, dual-edge sweep — explicit rainbow chrome against DESIGN’s Earned Color Rule. Decorative empty-state float and stacked uppercase eyebrows on expanded cards are secondary tells. Verdict: AI may have *written* the careful status system; AI almost certainly *added* the RGB lasers.

**Deterministic scan:** 17 findings (1 warning, 16 advisory) via `detect.mjs --json src/renderer`.
- `overused-font` (1): Inter in `index.html` — **false positive** (prescribed in DESIGN.md).
- `design-system-radius` (11): micro radii `1px`/`2px`/`4px`/`5px`/`7px` off documented scale (`6`/`10`/`14`/notch/full) in `main.css` + `components.css`.
- `design-system-font-size` (4): `9px` (×3) and `12px` — `9px` largely intentional per design prose (tiny mono chips); frontmatter mono is `10px`.
- `design-system-color` (1): `#1e1230` opencode tag border — likely intentional darkened agent border pattern (siblings use similar undocumented deep borders).
Detector did **not** flag the lasers (motion/gradient chrome is outside its rules). LLM and detector agree on token drift; LLM alone caught the brand-breaking rainbow chrome and a11y structure.

**Visual overlays:** Not available. Browser visualization SKIPPED — Electron-only app (`dev: electron .`), no localhost renderer, no mutable browser canvas for overlay injection in this harness.

## Overall Impression

AgentNotch’s *idea* and most of its *surface* are on-brand: quiet void shell, glanceable agent logos, attention only when earned, allow/deny as the emotional peak. The single biggest opportunity is to **delete continuous “busy” chrome** (rainbow lasers + always-loud working motion) so the product’s interrupt contract is legible again. Second: make high-stakes Allow honest (remote vs handoff) and make the keyboard/a11y surface match the visual polish of the permission strip.

## What's Working

1. **Status language is product-grade** — Bar copy (“Claude Code needs permission”), logo bright/dull, Sessions badge only when `attentionCount > 0`, tool humanization. Anti-red-dot-spam discipline matches PRODUCT.
2. **Inline attention actions are well-composed** — Allow/Deny, path, capped diff, remote badge, kbd chips, honesty hint for hook vs open-agent. Raycast/Linear-fluent users would nod here.
3. **Craft under the hood matches “precise”** — Fingerprint skip-rebuilds, fold state survival, feed scroll preserve, `prefers-reduced-motion` hooks, focus-visible rings (`--focus-ring`).

## Priority Issues

### [P0] Rainbow lasers violate brand and raise cognitive load
- **What:** `.notch-laser` / `.session-laser*` multicolor gradients, glow, hue-shift, dual edges on every working session (and the bar).
- **Why it matters:** Breaks Earned Color / no rainbow chrome; trains ignore-behavior by always looking busy; fights attention amber; is the “off component” pause for a design-fluent user.
- **Fix:** One quiet working cue — working-blue 1px edge or single soft bar underline; remove per-card rainbow; reduced-motion = static mono/blue strip if needed.
- **Suggested command:** `$impeccable quieter`

### [P1] Nested interactive + incomplete keyboard surface
- **What:** `.session-card` as `role="button"` wrapping Allow/Deny/options; history click-only; tabs not tab-pattern; no Escape-to-collapse.
- **Why it matters:** WCAG operable paths incomplete; screen readers get “button containing buttons”; keyboard users cannot reach history or exit cleanly.
- **Fix:** Header as expand control; detail actions outside button role; history keyboard parity; Escape collapses; true focus management when attention opens.
- **Suggested command:** `$impeccable harden`

### [P1] Permission Allow visual parity hides remote vs handoff
- **What:** Same `.btn-allow` for remote and `focus-only`; difference buried in `.approval-hint` / title.
- **Why it matters:** High-stakes action; users believe they approved in-notch when they only jumped to the agent. Undermines trust at the emotional peak.
- **Fix:** Remote: primary “Allow”; non-remote: “Open to allow” / secondary styling; promote hook install CTA when Claude is focus-only.
- **Suggested command:** `$impeccable clarify`

### [P2] Bar status color classes are dead code
- **What:** `statusClass()` adds `.working`/`.idle`/`.attention` to `#notch-status-text` with no CSS rules (unlike `.session-status-text.*`).
- **Why it matters:** Collapsed glance loses the status hierarchy the design system assumes.
- **Fix:** Mirror session status colors on bar text; keep non-color cues (icons + wording).
- **Suggested command:** `$impeccable colorize`

### [P2] Expanded card becomes a mini-IDE feed
- **What:** Live feed max-height ~420px + plan + prompt + tools + actions + usage + dispatch all at once.
- **Why it matters:** Violates status-before-chrome; first-timers lose the next action under transcript.
- **Fix:** Default card = status + next action; feed behind disclosure or last-N events.
- **Suggested command:** `$impeccable distill`

## Persona Red Flags

**Alex (Power User):** Lasers on N cards = peripheral noise; dispatch always visible on Settings/History; Cursor/Antigravity silently non-dispatchable; no Escape; wants quieter working state and denser keyboard map.

**Sam (Accessibility-dependent):** Nested `role="button"` cards; history not keyboard-reachable; 9px labels borderline; bar status color not implemented; reduced-motion still leaves full-width multicolor strip; toast-only Clear History confirm easy to miss.

**Jordan (First-timer):** Chevron opacity 0 until hover; first expand auto-opens card + laser + feed → unclear next action; Mac copy says ⌘ but kbd chips hardcode `Ctrl+`; Settings hook story buried.

**Project — Solo mid-task developer:** Glancing bar works if logos + wording suffice; laser steals glance without naming agent; trust breaks if Allow ≠ allow.

**Project — Interrupt-only trust:** Always-on working motion trains ignore; attention badge + auto-expand is correct — keep that, cut continuous rainbow.

## Cognitive Load

- **Failures:** Single focus, chunking (6 agent toggles), visual hierarchy (lasers = decoration as loud as attention), one-thing-at-a-time → **4 failures → high** when ≥2 agents working + expanded feed; **moderate** for pure glance + single attention card.
- **>4 options:** Agent enable list (6); unbounded question options; many-session dispatch select.

## Emotional Journey

Idle is excellent (calm peak). Working state is the emotional valley (demo-mode anxiety). Attention + permission strip is the product’s best peak — protect it from chrome competition. Completion (dull logo + quiet green) honors peak-end without celebration spam. Focus-only Allow is a trust dip at high stakes.

## Minor Observations

- `.notch-stats` markup still in HTML but CSS-hidden — dead surface.
- History detail uppercase labels stack more eyebrows.
- Settings six-agent toggle group could chunk better.
- Focus rings are consistently thought through — keep.
- Detector radius micro-steps: document or consolidate to DESIGN tokens.
- Toast ambient shadow is the only soft “premium float” — minor.

## Questions to Consider

1. If working state were **silent** (bright logo only) and only attention got motion/chroma, would “interrupt only when it matters” trust go up overnight?
2. Is the live activity feed a **second product** (transcript viewer) inside a status notch — should it collapse by default?
3. Would a Linear-fluent user ship a **pride-gradient progress beam** on a system utility?
4. Should **Deny** require friction and **Allow (remote)** feel instant — or is equal weight the moral design?
5. When five agents finish, is center prose needed, or does one dull logo row better honor Unobtrusive?
