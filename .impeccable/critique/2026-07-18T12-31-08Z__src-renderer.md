## Anti-Patterns Verdict

The interface does not look like a generic SaaS dashboard; it has a compact, system-tool character that fits the product. Its weaker visual choices are the continuously spinning/shaking pixel agents and spring/bounce easing, which compete with true attention states. The deterministic scan found five warnings: Inter from Google Fonts, three bounce/elastic motion findings, and one layout-transition finding. The remaining 20 advisory findings are mostly literal colors, small font sizes, and radii; they are design-system drift because the project lacks a DESIGN.md, not necessarily visible defects.

## Overall Impression

AgentNotch already has the hard part: one compact surface that combines multiple local agents. The single biggest opportunity is to make it an attention-triage tool rather than a detailed activity viewer: every interruption should say who needs what, how risky it is, and exactly what happens when the user acts.

## What's Working

- The unified session list, relative activity, tools, prompt, model, and duration provide unusually good local context without sending data anywhere.
- Status styling and labels distinguish working, finished, and attention states; the usage strip is compact and appropriately secondary.
- The visual language is restrained and technically appropriate: dark, dense, and quiet rather than dashboard-like.

## Priority Issues

### [P1] Attention does not reliably win the initial view

**Why it matters:** The product promise is that a user can identify the agent needing them in under a second. The initial renderer selects and expands the first session when no session is already selected, even if a later card has a question or permission. In the inspected preview, Claude’s working card opens first while Antigravity’s question is lower in the list.

**Fix:** Sort attention sessions above working and completed sessions; on initial load and every update, expand the highest-priority attention card. Place its question/permission and a single clear next step above all activity detail. Add a small “2 need you” queue control when there are multiple attention sessions.

**Suggested command:** $impeccable polish

### [P1] Action labels are misleading and do not close the loop

**Why it matters:** “Allow,” “Deny,” and answer options appear to be remote actions, but the implementation only focuses the external agent. Users can believe a risky action was approved when it was not; failures are only logged to the console.

**Fix:** Rename these controls to “Open approval in Claude,” “Open question in Antigravity,” etc. After activating, show a visible pending handoff state with the destination and a “Still waiting?” retry/reveal option. If direct approval becomes supported, use a distinct label such as “Approve now” only then.

**Suggested command:** $impeccable clarify

### [P1] Monitoring trust is under-specified

**Why it matters:** A status monitor is only useful when users know whether its observation is current. `formatRelativeTime` presents missing timestamps as “just now,” masking unavailable or stale data. The empty view has a detection list, but active sessions lack source/last-seen confidence and watcher health.

**Fix:** Introduce explicit states: “live,” “last update 4m ago,” “watcher unavailable,” and “agent source not detected.” Show the source and last event on expanded cards; add a concise health item in Settings. Never map unknown time to “just now.”

**Suggested command:** $impeccable harden

### [P1] Keyboard and assistive-tech use break on core session cards

**Why it matters:** Session cards are clickable `div`s, so they cannot be reached or expanded using a keyboard. There are no visible focus styles, and there is no reduced-motion fallback despite indefinite spinning, shaking, pulsing, and bouncing. This is especially problematic for an always-on utility.

**Fix:** Make each card header a real button with `aria-expanded` and keyboard operation; preserve nested action buttons. Add high-contrast `:focus-visible` outlines, live announcements for new attention, non-color state icons/text, and a `prefers-reduced-motion` rule that removes perpetual motion.

**Suggested command:** $impeccable audit

### [P2] The expanded view optimizes for inspection before triage

**Why it matters:** Prompts, recent tools, activity, plans, models, and usage are useful after deciding where to act, but the first open card consumes most of the panel. Important attention sessions end up visually lower and less readable.

**Fix:** Use a two-level layout: an “Needs you now” zone containing only agent, task, blocker, risk, and action; then an “In progress” compact list. Move prompt, plan, and tools behind an explicit “Details” disclosure. Make usage limits a secondary settings/status surface unless a limit is close.

**Suggested command:** $impeccable layout

## Persona Red Flags

**Alex (Power User):** Must inspect cards manually to find an urgent session if the first card is merely working. Cannot keyboard-expand a session; answer shortcuts require remembering Ctrl+1–9 and do not provide a navigation path between multiple attention items.

**Sam (Accessibility-Dependent User):** The card expansion is click-only; focus indicators are absent, motion has no reduced-motion alternative, and the collapsed strip relies heavily on tiny colored pixel icons.

**Riley (Stress Tester):** A watcher with no valid timestamp looks “just now,” and a failed approval/focus request leaves no visible error or recoverable action. The monitoring surface can silently become untrustworthy.

## Minor Observations

- Normalize product names everywhere: decide whether the user sees Antigravity or Gemini, and Grok or Grok Build.
- Move the task-dispatch composer behind a deliberate “New task” action or give it an explicit shortcut; it is a separate job from passive monitoring.
- Provide granular notification policy: only permissions/questions by default; completed turns should be quiet or bundled.
- Replace the spring/bounce motion with short state transitions and suppress all perpetual motion when reduced motion is enabled.
- Add a DESIGN.md or move literal colors, sizes, and radii into tokens to stop visual drift.

## Questions to Consider

- Is the primary promise “never miss a blocker,” or should the app also become the place users start/dispatch work? These are competing interaction models.
- Would users trust a status only when it is live, or should stale observations trigger a separate “check agent” warning?
- Which agents can realistically support direct approval/answer actions, and which should always be handoffs?
